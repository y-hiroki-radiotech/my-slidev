# Student Review

## Overview

学習教材（Markdownファイル）を初学者の視点で分析し、理解が難しい箇所や疑問が生じそうなポイントを特定するスキル。教材の改善点を洗い出し、詳細なレビューレポートを生成する。

**主な用途:**
- 講義資料作成後の品質チェック
- チュートリアルのわかりやすさ検証
- 既存教材の改善点特定

## Workflow

1. **ファイル読み取り**
   - 指定されたMarkdownファイルを読み取る
   - 内容全体を把握し、対象学習者のレベルを判断

2. **初学者視点での分析**
   - 各セクション、段落、説明を初学者の目線で読む
   - 理解が難しい箇所、疑問が生じそうなポイントを特定
   - 専門用語、前提知識、説明の飛躍などに注目

3. **疑問点の整理**
   - 特定した各疑問点について以下を分析:
     - どの箇所で疑問が生じるか
     - なぜその箇所がわかりにくいか
     - どう改善すればわかりやすくなるか
     - その知識が実務でどう役立つか

4. **レポート生成**
   - `reviews/` ディレクトリが存在しない場合は作成
   - 詳細レポートをMarkdown形式で作成
   - `reviews/` ディレクトリに元のファイル名で保存
   - 例: `docs/lecture-01.md` → `reviews/lecture-01.md`
   - 例: `slides.md` → `reviews/slides.md`

## Output Format

生成されるレビューレポートは以下の構造を持つ:

```markdown
# 学生視点レビュー結果：[元のファイル名]

## 全体評価
[教材全体の印象、主な強みと改善の余地がある領域]

## 疑問点 1: [セクション名または内容の要約]
**箇所:** [該当する文章や説明を引用]

**なぜ疑問が生じるか:**
[初学者がこの箇所で躓く理由を説明]

**改善提案:**
[具体的な書き直し案や追加すべき説明]

**現場での活用:**
[この知識が実際の業務やプロジェクトでどう使われるか、具体例を交えて説明]

## 疑問点 2: ...
[同様の形式で続ける]

## まとめ
[レビュー全体のまとめと優先的に改善すべきポイント]
```

## Review Guidelines

### 初学者視点で考えるポイント

**1. 専門用語と前提知識**
- 説明なしに専門用語を使っていないか
- 読者が知っていると仮定している前提知識はないか
- 用語の初出時に定義や説明があるか

**2. 説明の論理的な流れ**
- 段落間、セクション間のつながりが明確か
- 説明に飛躍がないか（AからCに飛んでBが抜けている等）
- 「なぜそうするのか」の理由が説明されているか

**3. 具体例とイメージのしやすさ**
- 抽象的な説明だけでなく具体例があるか
- 比喩や例え話は適切でわかりやすいか
- コード例がある場合、初学者が理解できる複雑さか

**4. 情報の量と密度**
- 一度に多くの概念を詰め込みすぎていないか
- 重要なポイントが埋もれていないか
- 段階的に理解を深められる構成か

### 現場での活用例の書き方

実務での活用場面を説明する際は、以下の要素を含める:

**1. 具体的な場面設定**
- 「Webアプリ開発では...」「データ分析の現場では...」のように具体的に
- 初学者が想像しやすい身近な例（TwitterやAmazonなど）を活用

**2. 実例の提示**
- 「例えば、ログイン機能では...」のように具体的な機能やシステムに紐付ける
- 学習している概念が実際にどう使われているかを明示

**3. 学習のモチベーション**
- 「この知識なしには〜ができない」といった必要性
- 「実際のプロジェクトでは頻繁に使う」といった頻度や重要性

**悪い例:**
> 変数は重要な概念です。プログラミングで使います。

**良い例:**
> 実際のWebアプリ開発では、ログイン中のユーザー名、ショッピングカートの商品リスト、画面に表示するメッセージなど、あらゆるデータを変数で管理します。例えばTwitterでは、あなたのユーザー名、フォロワー数、投稿内容すべてが変数として扱われています。変数の理解なしにはどんなアプリも作れません。

## Usage Example

```
/student-review docs/lecture-01-variables.md
```

Claude will:
1. Read the file
2. Analyze from beginner's perspective
3. Create `reviews/` directory if it doesn't exist
4. Generate `reviews/lecture-01-variables.md` with detailed feedback

### Output Location

すべてのレビューレポートは `reviews/` ディレクトリに保存されます:

```
reviews/
├── lecture-01-variables.md    # docs/lecture-01-variables.md のレビュー
├── slides.md                  # slides.md のレビュー
└── section-overview.md        # pages/section-overview.md のレビュー
```
